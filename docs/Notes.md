## 实验报告规范：
报告至少包含以下部分：

- 实验目的

    - 用简短概括性的一句或几句话讲讲你所理解的本次实验想要你学会什么。

- 实验过程：

    - 记录代码的设计思路、实现步骤、特别是遇到的问题及其解决方案。我们鼓励讲述 debug 的过程，以及解决 bug 的方法。 

- 实验结果

    - 贴出实验样例通过的截图，简要解释测试样例测试了什么内容。

- 思考题

    - 请附上你对每个实验末尾 **实验报告要求** 一节所有问题的回答。

- 心得体会：

    - 我们十分鼓励表达你真实的感受与想法，而不是官方而正确的废话。你既可以反思实验中遇到的挑战和收获，也可以表达对这次实验设计的吐槽与攻击，也可以表达对助教工作的不足与未来的改进，请尽可能畅所欲言。

我们不希望实验报告成为程式化的重复性任务。为了减轻大家的负担，大家 **不需要在实验报告上卷字数，页数，美观** ，只要符合基本要求都能拿到高分数。但还是希望大家表达真实想法。

## 代码查重：
1. **代码查重机制**：
    - 课程组将使用代码查重工具，设置有适当的重复率阈值，以防同学之间或网上资料的抄袭。我们会将往届的代码以及学长学姐们已经开源的代码加入查重库，希望大家不要直接复制。
   
2. **对查重不合格的处理**：
    - 第一次查重不合格，**本次实验计 0 分**；第二次查重不合格，**课程总评最高不超过 59 分**。我们会在每一次提交代码后进行查重，并和重复率较高的同学沟通。


## 作业提交要求：

在学在浙大分别提交 **代码/code** 和 **报告/report**：

- 代码：提交 submit 文件夹压缩包，命名格式为：学号_lab?.zip，如 3230123456_lab0.zip

- 报告：提交实验报告，命名格式为：学号_lab?.pdf，如 3230123456_lab0.pdf

## 验收标准：
- 在验收时，助教会提出与实验相关的问题，如代码设计、实现细节及遇到的问题等。

- **回答问题不通过则验收不通过**，但仍然可以下次继续验收，直到通过为止。并且不会有刁钻问题，请无需有压力。

## Issue管理与激励措施：
- 鼓励通过 ZJUGit 的Issue功能提问，而不是依赖 P2P 直接交流。这样不仅有助于建立共享的知识库，还可以避免重复解答问题。

- 通过 Issue 进行讨论和提问的积极性高的同学可以获得额外奖励。
- 提交 Issue 的标准见 [提问方式](./issue.md)

## 关于向助教提问

我们推荐大家使用 Issue 进行提问来形成共享知识库，但并不代表大家不能私信询问助教，但是请注意 **并非所有的问题助教都曾碰到过**，因此请大家提问时注意以下几点：

- 提供完整的问题截图和描述，如果问题方便复现可以详细描述发现问题的过程
- 对于编译器报错/错误代码等错误，**助教大概率也并不知道原因**，因此在询问前可以善用搜索引擎，在 CSDN, Github, Stack Overflow, Stack Exchange 等相关技术网站上大概率能找到解决方案。
- 我们鼓励大家在私信解决问题后，将问题的发现和解决的全过程总结发在课程仓库的 Issue 区供其他同学参考，这里也附上往年同学碰到过的问题和解答：[Sp24](https://git.zju.edu.cn/zju-sys/sys1/sys1-sp24/-/issues), [Sp23](https://git.zju.edu.cn/zju-sys/sys1/sys1-sp23/-/issues)。
